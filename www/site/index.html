Eo9

A capability-secure operating system built on the WebAssembly Component Model.

Source on GitHub Read the spec

Status: Eo9 is in early, active development. Nothing described here is stable, and every concrete design choice is provisional. The spec is the source of truth.

What Eo9 is

Eo9 is an operating system built on language-theoretic principles for security, performance, and composability. Programs are distributed as WebAssembly Component Model components rather than native binaries, and the OS itself is, at its core, a set of typed APIs plus a compiler for programs that use them.

A WebAssembly component's only channel to the outside world is its imports — there are no syscalls and no ambient I/O. The import set therefore is the capability set: everything a program can do is statically enumerable from its imports before it ever runs. Eo9 does not rely on hardware privilege domains for isolation; it is secure by construction, with a small, explicit trusted computing base (the compiler, the root scheduler, and the hardware-root capabilities held by the OS core).

Composition is the whole interface

Granting, attenuating, and replacing capabilities are all the same operation: composing providers onto a program. $ composes a provider into a consumer, satisfying and sealing the imports it matches; & extends one provider environment with another, so reusable environments are ordinary values. Dropping a capability (net.none), refusing it (net.deny), or bounding a whole composition to an allow-list (only) are part of the same small algebra. 

> net.none $ browser                        # browser observes "no network"
> time.monotonic-stub & virtualnet $ app    # a packaged environment, applied to app
> only eo9:time,eo9:fs $ cruncher --input data.bin   # allow-list; anything else is a compose-time error

Arguments are named and typed — there is no untyped argv — and a program reports its outcome in its own typed vocabulary rather than a numeric exit code.

Virtualization and sandboxing by substitution

Every OS API a program uses can be substituted at composition time: a virtual filesystem, loopback-only networking, a frozen clock, seeded entropy. There is no separate hypervisor mechanism — a sandbox is just a different set of providers. Because providers are compiled together with the program, substitution has zero overhead: a layer that does nothing compiles away entirely.

Deterministic execution

Compose a program with a deterministic environment — fs.memfs & time.frozen & entropy.seeded — and its runs become reproducible. Scheduling is fuel-metered, so the CPU interleaving is fixed, and with deterministic providers the completion order is too: deterministic tests go all the way down.

Performance

Programs are compiled from bytecode to the native architecture with their providers inlined. There is no kernel-mode/user-mode boundary and no context switch except scheduler switches between programs, and no MMU or nested address translation is needed for isolation. One honest caveat: on MMU-less targets, linear-memory safety is paid for with the bounds checks the optimizer cannot prove away.

eofs, the native filesystem

Eo9 bundles its own filesystem, eofs, and it is just a provider: it imports the disk API and exports the filesystem API, so the same component runs over a real block device on bare metal and over a file-backed or in-memory disk under usermode Eo9. It is copy-on-write and never overwrites in place, with atomic root flips — crash consistency by construction, no fsck. It is a Merkle tree throughout, so every file and directory carries a precomputed content hash up to the root: integrity checking, cheap change detection, and hash-guided incremental backup. Snapshots are retained roots, and block compression is on by default.

Bare metal and hosted

Eo9 targets bare metal — AArch64, RISC-V, and x86-64, including MMU-less hardware — and, because it is fundamentally a WebAssembly compiler plus a set of APIs, the same system also runs as an ordinary usermode program on an existing OS (or on Eo9 itself).

Roadmap

The MVP being built in the open right now aims at:

  • the core OS API specs (execution, disk, filesystem, net, text, messages, entropy, time, perf);
  • a usermode eo9 binary backed by standard *nix APIs;
  • bootable QEMU images for AMD64, AArch64, and rv64gc — headless programs and boot-to-shell;
  • eosh, a shell built around the composition operators;
  • usermode and in-QEMU test suites.

Progress, plans, and the full design live in the repository: github.com/wyager/eo9.
