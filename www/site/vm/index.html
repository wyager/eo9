What this is

The buttons above drive a single ~1 MB WebAssembly module: the same wasmtime the native Eo9 runtime and the bare-metal kernel embed, compiled for wasm32, with its Pulley interpreter executing pre-compiled images of unmodified Eo9 components. The page's JavaScript only provides this output box; no Eo9 semantics are re-implemented in JS.

  • hello + add — the kernel's seed component (the first thing Eo9 ever ran on bare metal), executed by Pulley inside your browser.
  • fuel metering — the same call with wasmtime's fuel accounting on, the mechanism Eo9 uses to meter CPU.
  • entropy.seeded — an unmodified Eo9 provider stub using the Component Model async ABI: its configure and get-u64 are async-lifted calls, executed here without fibers (wasm32 has none) via Eo9's fiberless execution path. Same seed, same sequence as native Eo9 and the bare-metal kernel.
  • program store — the second row fetches real Eo9 example programs (pre-compiled Pulley images served as static files) and runs them against browser root providers: text → this terminal, time → the browser clocks, entropy → crypto.getRandomValues. Typed arguments in, a typed outcome out — the same contract as native eo9 run.
  • park the VM / read a line — genuinely blocking operations: the VM (and any guest awaiting time.sleep or read-line inside it) is parked on a real browser timer or on the input box below the terminal, using JavaScript Promise Integration (JSPI) to suspend the whole VM and resume it. These need a JSPI-capable browser (current Chrome/Edge; the page says so if yours is not). The sleepy canary uses the Component Model's stackful async lift, which still needs a fiber backend — the page reports that limitation honestly rather than hiding it.

One disclosure, so the name of the page doesn't overclaim: the three embedded demos in the first row import nothing at all — they are evidence that the real engine and the real component ABI run in your browser, not a demonstration of capability mediation. The program-store row is where imports are actually linked (text, time, entropy), and the /try page is where granting and revoking capabilities is the demo.

What is not here yet: the shell (eosh), the capability algebra, and on-page composition. Those are the next milestones for this page; the /try page remains the polished demo in the meantime.
