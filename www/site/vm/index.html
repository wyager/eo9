What this is

The buttons above drive a single WebAssembly module (about 1.8 MB compressed on the wire): the same wasmtime the native Eo9 runtime and the bare-metal kernel embed, compiled for wasm32, with its Pulley interpreter executing unmodified Eo9 components — and the same Cranelift-based compile layers the bare-metal kernel uses, so compositions typed at the prompt are compiled right here in the page. The page's JavaScript only provides this output box and your keyboard; no Eo9 semantics are re-implemented in JS, and no server is involved after the page loads.

  • hello + add — the kernel's seed component (the first thing Eo9 ever ran on bare metal), executed by Pulley inside your browser.
  • fuel metering — the same call with wasmtime's fuel accounting on, the mechanism Eo9 uses to meter CPU.
  • entropy.seeded — an unmodified Eo9 provider stub using the Component Model async ABI: its get-u64 draws are async-lifted calls, executed here without fibers (wasm32 has none) via Eo9's fiberless execution path. Same seed, same sequence as native Eo9 and the bare-metal kernel.
  • program store — the second row fetches real Eo9 programs (the example programs plus all twelve coreutils, pre-compiled Pulley images served as static files) and runs them against browser root providers: text → this terminal, time → the browser clocks, entropy → crypto.getRandomValues, and an in-memory eo9:fs living inside the blob. Typed arguments in (the path-taking tools take positional, variadic paths, e.g. cat /welcome.txt /docs/about.txt), a typed outcome out — the same contract as native eo9 run.
  • the eosh shell — the third row boots the real Eo9 shell, unmodified. It resolves programs from /bin, spawns them through the same typed eo9:exec surface the native CLI and the bare-metal kernel use, and the capability algebra works at the prompt: only eo9:text $ echo --text hi genuinely narrows what the child can import (a program needing a sealed-away capability is refused before it runs), and entropy.seeded $ rng --count 3 is fused by the real algebra and runs deterministically.
  • compiler in the VM — the blob carries the same Cranelift-based compile layers the bare-metal kernel uses for on-target codegen, emitting Pulley bytecode: a composition typed at the eosh prompt (e.g. entropy.seeded $ rng) is fused by the real component algebra and compiled inside your browser, with no server involved — the same compose → compile → run story as native Eo9 and bare metal.
  • park the VM / read a line — genuinely blocking operations: the VM (and any guest awaiting time.sleep or read-line inside it) is parked on a real browser timer or on the input box below the terminal, using JavaScript Promise Integration (JSPI) to suspend the whole VM and resume it. These need a JSPI-capable browser (current Chrome/Edge; the page says so if yours is not). The sleepy canary — the kernel's stackful-async sleep test — runs here too, awaiting a real browser timer in the middle of a guest call and reporting the elapsed time it measured.

One disclosure, so the page doesn't overclaim: the three embedded demos in the first row import nothing at all — they are evidence that the real engine and the real component ABI run in your browser, not a demonstration of capability mediation. The mediation demos are the rows below them: the program store links real imports (text, time, entropy, fs), and the shell's only / $ are the real algebra doing real attenuation and composition.

What is not here: persistence (the in-blob filesystem is in-memory and resets on reload), networking, and native-speed execution — everything runs through the Pulley interpreter, so this page is about fidelity, not performance. The /try page is a smaller, separate demo that runs transpiled components on the browser's own engine.
