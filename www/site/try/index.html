Things to try

hello --name browser --excited true cruncher --seed 7 --rounds 100000 readwrite --path /notes/hi.txt --contents "written from wasm" revoke time hello --name browser --excited true

Run cruncher twice — it imports nothing at all, so it is pure compute and the digest is the same on every machine. After readwrite, files shows what it wrote to the in-page filesystem. And after revoke time, hello is refused before anything runs, because a required import is no longer satisfiable — that refusal, not a runtime permission check, is Eo9's security model in one line. grant time puts it back.

What is real here

The programs behind this page are the actual example components from the Eo9 repository — hello, outcomes, cruncher, readwrite — the same .wasm artifacts the usermode eo9 binary runs. At build time they are translated 1:1 from the Component Model binary format into ES modules plus core wasm (using jco's transpiler), and your browser's WebAssembly engine executes them here.

The part that is not Eo9 is deliberately small: a hand-written browser host on this page plays the role the eo9 usermode runtime plays natively. It provides the root capabilities the programs import — eo9:text is wired to the terminal above, eo9:time to your browser's clock, eo9:fs to an in-memory filesystem that lives in the page — and it applies the loader rule: a program's imports are checked against what the host provides, and a missing required import means the program is refused before it is instantiated. Use grants, grant, and revoke to see that from the outside. To be precise about where the security lives: the polite refusal text is printed by this page's launcher, but the enforcement is that a revoked capability is simply never put in the import object handed to the module — going around the launcher doesn't get you the capability, it just gets you a less helpful error.

The prompt above is not eosh. It is a launcher with a handful of commands, and there is no Eo9 compiler on this page: the components are translated to ES modules at site build time and run on the browser's own engine. The real Eo9 shell, the stub providers (entropy.seeded, fs.memfs, …), the composition algebra ($, only, …), and the real engine-plus-compiler stack do run in your browser — on the /vm page, where the whole Eo9 runtime is compiled to WebAssembly and compositions typed at the eosh> prompt are compiled inside the page. This page stays deliberately small: it shows the capability rule itself — imports granted, revoked, and refused — with the least machinery possible.

Browser support

hello, outcomes, and cruncher run in any modern browser. readwrite has an async main, which the transpiled form drives with JavaScript Promise Integration (JSPI); recent Chromium-based browsers support it, and the page tells you if yours does not. Everything runs client-side — the server only hands out static files, and nothing you type leaves your machine.
